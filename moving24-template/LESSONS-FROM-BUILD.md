# Уроки и правила из реальной сборки Moving24

Это не теория. Это правила, выведённые из **реальной работы над Moving24 и обратной связи владельца**: каждая правка, которую пришлось сделать, и причина, которая к ней привела. При клонировании под новый бизнес читай это ПЕРВЫМ - оно экономит недели ошибок.

Формат: **ПРАВИЛО -> ПОЧЕМУ (что реально случилось) -> КАК ПРИМЕНЯТЬ**.

---

## 1. Копирайтинг и язык

### 1.1 Никаких длинных тире. Только дефис `-`.
- **Почему:** жёсткое требование владельца. Длинное тире `-` и en-dash проскакивали от носителей и из ИИ-текстов постоянно (в диапазонах дней `E-R`, в заголовках, в присланных правках).
- **Как применять:** любое `-`/`-` заменять на дефис `-` или переписывать фразу. В `validate.py` уже стоит проверка на em/en-dash - гоняй её перед каждой сборкой. Когда носитель/клиент присылает текст с тире - молча конвертируй.

### 1.2 Никаких упоминаний страховки.
- **Почему:** жёсткое правило. Присланный носителем «идеальный» разбор главной был построен вокруг «kindlustatud veod / страховая сумма» - пришлось всё это вырезать, хотя текст был красивый.
- **Как применять:** не писать «застраховано / kindlustus / insurance / vakuutus / hüvitame kahju», даже если источник хороший. Ответственность формулировать без слова «страховка» (напр. «vastutame meie tegevusest põhjustatud kahju eest»). Проверяй grep-ом по всем языкам.

### 1.3 Только реальные факты. Никаких выдуманных цифр и плейсхолдеров.
- **Почему:** тот же разбор носителя содержал `[X] €/ч`, «отвечаем за 2 часа», страховые суммы - плейсхолдеры под реальные данные. На живой сайт такое ставить нельзя.
- **Как применять:** нет реальной цифры - не пиши цифру. Оставляй честные не-числовые формулировки («фиксированная цена заранее»). Отзывы - только настоящие, дословно, никогда не выдумывать (это отдельное жёсткое правило).

### 1.4 Нативное качество, один язык - мастер.
- **Почему:** владелец гонял тексты через носителей и требовал, чтобы читалось «как написал носитель», а не как перевод. У Moving24 RU - мастер, ET/EN/FI - нативные адаптации.
- **Как применять:** пиши на языке-мастере, остальные - нативно адаптируй (не дословный перевод). Детальные правила по каждому языку - в `playbooks/LANG-*.md`. Принципы ET: единая teie-форма, идиома вместо канцелярита, хлёсткие не-сужающие заголовки, единая терминология (`pakkematerjalid`, не `pakendimaterjalid`).

### 1.5 Слова не переносятся уродливо; заголовки не «сужают».
- **Почему:** правили «висячие» слова (`text-wrap:pretty`, nbsp), и меняли сухие/сужающие заголовки на живые (напр. `Kolimised igas mahus` -> `Ühest toast terve laoni`, `Meid usaldatakse` -> `Mida kliendid meist räägivad`).
- **Как применять:** заголовок должен звать/интриговать, а не описывать «что это». Проверяй перенос последнего слова на мобиле.

## 2. Трекинг и Google Ads - святое, не трогать кабинет

### 2.1 GTM-контейнер и рекламный кабинет НЕ меняем. Сайт подстраивается под них.
- **Почему:** самое жёсткое правило владельца («никто рекламный кабинет менять не будет, в этом была вся задумка», ~160 EUR/день). Кабинет и триггеры настроены под события старого WordPress-сайта. Новый статический сайт **воспроизводит те же события**, чтобы конверсии считались без единого изменения в GTM.
- **Как применять:** событие успеха формы - это `dataLayer.push({event:'ajaxComplete', ...})` в форме WPForms (admin-ajax.php, statusCode 200, wpforms-confirmation) - дословно, чтобы существующий триггер сработал. Не «улучшать» и не переименовывать события.

### 2.2 `stopPropagation()` на tel:/mailto: убивает конверсию звонка. Не добавлять.
- **Почему (главный грабль дня):** конверсии звонков/имейлов в GTM висят на нативном триггере Click (`gtm.linkClick`). Ранее добавленный в обработчик `e.stopPropagation()` (пытались лечить iOS-подсказку «автонабор») **молча глушил** нативный слушатель GTM - `gtm.linkClick` не срабатывал, звонки не считались. Форма работала (у неё свой кастомный ивент), а телефон - нет. Диагностировали через dataLayer на живом сайте.
- **Как применять:** обработчики tel:/mailto: только пушат событие и НЕ вызывают stopPropagation. Если появится iOS-подсказка «автоматический вызов» - лечится в GTM (снять «Wait for Tags» на триггере), а не на сайте.

### 2.3 Событие конверсии - только при реальном успехе.
- **Почему:** `ajaxComplete` пушится только внутри `if(sent)` при HTTP 2xx от `lead.php`. При ошибке/сети/демо - не пушится (иначе мусорные конверсии).
- **Как применять:** не пушить конверсию «на клик отправки» - только после подтверждённого 200 от бэкенда.

### 2.4 Consent Mode v2 default=denied; GTM грузится только на боевом домене.
- **Почему:** гейт по hostname (`/(^|\.)domain$/`) - GTM не грузится на localhost (поэтому в локальных тестах его нет, это норма). Consent default=denied до баннера.
- **Как применять:** при клоне поменять домен в гейте и (см. секцию трекинга гайда) ID контейнера. Не путаться, что «на localhost GTM не видно» - это by design.

### 2.5 `phone_link_klick` - написание нарочно такое.
- **Почему:** так называется событие/триггер, на который завязан кабинет. «klick» не опечатка.
- **Как применять:** не «исправлять» на klik/click.

## 3. Форма и UX

### 3.1 Условные поля формы скрыты по умолчанию (`style="display:none"`), иначе «вспышка».
- **Почему (грабль дня):** поля, зависящие от выбранной услуги, были видимы в HTML и прятались JS-ом уже после загрузки. На мобиле форма грузилась высотой **891px**, потом схлопывалась до **466px** - скачок 425px, выглядело как «форма мелькнула/пропала». Измерили дельту на живом сайте.
- **Как применять:** всё, что скрывается скриптом при инициализации, должно быть скрыто уже в HTML (инлайн `display:none`). JS показывает через `display:''` (очищает инлайн). Никаких «показать в HTML -> спрятать скриптом».

### 3.2 Бургер появляется РОВНО там, где исчезает десктоп-меню.
- **Почему (грабль дня):** меню пряталось на `max-width:1020px`, а бургер появлялся только на `760px`. В зазоре 761-1020px навигации не было вообще. Проверили через CSSOM.
- **Как применять:** скрытие меню и показ бургера (+ открытие мобильного оверлея) - на ОДНОМ брейкпоинте. Проверять именно диапазон между брейкпоинтами.

### 3.3 Никаких «бегущих» цифр.
- **Почему:** владелец жёстко: «цифры нигде на сайте не должны бегать» (капсом). Счётчик анимации переездов удалён полностью - число ставится статически.
- **Как применять:** счётчики/статистика - статичные значения, без count-up анимаций.

## 4. Сборка, деплой, миграция

### 4.1 Не заливать новый сайт ПОВЕРХ старого. Сначала очистить веб-рут.
- **Почему (грабль дня):** новый статический сайт распаковали поверх старого WordPress в `htdocs`. Apache отдавал старый `index.php` раньше нового `index.html` - показывался старый сайт, хотя новые файлы уже лежали.
- **Как применять:** wipe `htdocs` -> распаковать зип -> положить `config.php`. Старый сайт заранее в отдельную папку/снапшот для отката.

### 4.2 На проде снять `noindex` из `.htaccess`.
- **Почему:** на staging стоит `X-Robots-Tag: noindex` (чтобы staging не индексировался). Если залить как есть на прод - весь сайт выпадет из Google.
- **Как применять:** prod-версия `.htaccess` без noindex-строки (в шаблоне сборка prod-зипа уже её вырезает). Проверять `curl -I` на отсутствие noindex.

### 4.3 `config.php` никогда не в зипе и не в git. Dev-мусор тоже не на прод.
- **Почему:** `config.php` содержит Gmail app-password. Плюс в зип случайно попадали dev-файлы (`.claude`, `_ai/GUIDE.md`, `build.txt`, `.nojekyll`) и торчали наружу (`/_ai/GUIDE.md` -> 200, внутренняя дока).
- **Как применять:** `build_zip.py` исключает `config.php` + dev-мусор (список EXCLUDE в скрипте). Секреты - только на сервере. После деплоя проверять, что `/config.php` отдаёт пустое тело, а dev-путей нет.

### 4.4 Каждое изменение: `validate.py` -> `build_zip.py`. Помни про «разрыв деплоя».
- **Почему:** правки живут только в зипе, пока владелец не зальёт его на хостинг. Много раз он смотрел старую версию, потому что не перезалил.
- **Как применять:** после правок - валидируй и пересобирай зип. Явно говори «перезалей зип, иначе изменений не видно».

## 5. Миграция и SEO (перенос со старого сайта)

### 5.1 301-редиректы со всех старых URL на новые. Топовые URL сохранять как есть.
- **Почему:** старые проиндексированные URL брали из GSC (Performance -> Pages). Топовая страница (`vyvoz-musora`, 8654 показа) в новой структуре осталась на том же URL - редирект не нужен, трафик сохранён. Остальные (garbage-collection -> junk-removal и т.д.) - 301.
- **Как применять:** снять список старых URL из GSC, смапить на новые, прописать `RedirectMatch 301` в `.htaccess`. Где старый URL совпадает с новым - не редиректить (иначе петля). Старые sitemap-пути тоже 301 на новый sitemap.

### 5.2 GSC: удалить старые sitemap, добавить новый. «Couldn't fetch» часто = просто ещё не дочитал.
- **Почему:** старые Yoast-сайтмапы (`page-sitemap.xml`, `post-sitemap.xml`) после сноса WP отдают 404 - GSC пишет ошибку. Новый `sitemap.xml` сразу после отправки тоже показывал «Couldn't fetch» при полностью рабочем файле - это временный статус Google.
- **Как применять:** удалить старые сайтмапы из GSC, отправить `sitemap.xml`, подождать 24-48ч. Не пере-отправлять по кругу. Проверить, что sitemap отдаётся как `text/xml`, 200, и Googlebot его берёт.

### 5.3 Sitemap: свежий `lastmod`, hreflang-альтернативы, только каноничные 200-URL.
- **Почему:** `lastmod` был устаревший - обновили на дату правок (толкает переиндексацию). В sitemap есть `xhtml:link` hreflang для всех языков.
- **Как применять:** в sitemap только страницы, отдающие 200 (никаких старых/404/редиректных), с актуальным lastmod и hreflang-набором.

### 5.4 Telegram/соцсети кешируют OG. Обновляется через @WebpageBot.
- **Почему:** после переезда Telegram показывал старую OG-карточку - это кеш, а не ошибка сайта (проверили: живой HTML отдаёт новый OG).
- **Как применять:** для сброса кеша превью в Telegram - отправить URL боту @WebpageBot.

## 6. Процесс и проверка

### 6.1 Проверять на ЖИВОМ, детерминированно, а не «на глаз».
- **Почему:** ресайз окна в браузере не менял реальный `innerWidth` - брейкпоинты проверяли через **CSSOM** (в каком медиа лежит правило), трекинг - через **dataLayer** (что реально пушится при клике), доступность/редиректы/заголовки - через **curl**. Это точнее скриншотов.
- **Как применять:** для тонких вещей (адаптив, трекинг, редиректы, заголовки) - инструментальная проверка, а не визуальная.

### 6.2 Присланный носителем/клиентом текст - фильтровать через правила.
- **Почему:** «идеальный» разбор носителя содержал страховку, плейсхолдеры и en-dash - применили только безопасную языковую часть, остальное вырезали, и явно объяснили владельцу почему.
- **Как применять:** любой внешний текст прогонять через правила 1.1-1.4 перед вставкой. Расхождения проговаривать владельцу, не применять молча.

### 6.3 Когда правило владельца конфликтует с «улучшением» - правило владельца выше.
- **Почему:** несколько раз «здравое улучшение» (тронуть GTM, оставить страховку, длинный H1 по SEO) конфликтовало с прямым указанием владельца. Каждый раз выигрывало указание владельца.
- **Как применять:** если хочется сделать «правильнее по учебнику», но это против явного правила клиента - сначала спросить, а не сделать.

## 7. Письмо-заявка (lead.php) и Google Maps

### 7.1 Телефон и email в письме - кликабельные, сообщение клиента - выделено.
- **Почему:** заявка приходит менеджеру на телефон. Владелец просил, чтобы по телефону можно было тапнуть-позвонить, по email - ответить, а текст клиента не терялся среди деталей.
- **Как применять:** телефон - `<a href="tel:...">` (в href только цифры и `+`), email - `<a href="mailto:...">`. Сообщение клиента - отдельный выделенный блок (жёлтая подложка, рамка слева, крупный шрифт), а не строка в общем списке.

### 7.2 Дата в письме: ДД.ММ.ГГГГ + день недели.
- **Почему:** `<input type="date">` шлёт ISO `ГГГГ-ММ-ДД` - в письме читалось «с года», неудобно. Плюс менеджеру полезно сразу видеть день недели.
- **Как применять:** на бэке переформатировать в `ДД.ММ.ГГГГ (день недели)`, день недели считать из даты (`date('w')` -> русское название).

### 7.3 Картинки (карта, Street View) в письме - встраивать inline (CID), не ключом в URL.
- **Почему:** если вставить `<img src="...staticmap?...&key=KEY">`, ключ светится в письме и картинка требует «загрузить изображения». Мы тянем картинку на сервере и встраиваем через `addStringEmbeddedImage` (cid:) - ключ остаётся на сервере, картинка видна сразу.
- **Как применять:** серверный `fetchUrl` (cURL + фолбэк) -> `addStringEmbeddedImage($png,$cid,...)` -> в HTML `<img src="cid:...">`. Проверять сигнатуру (PNG/JPEG), иначе не вставлять.

### 7.4 Google Maps ключ: ДВА разных ограничения. «HTTP referrers» ломает серверный вызов.
- **Почему (грабль):** сперва `REQUEST_DENIED` из-за «API restrictions» (ключу не разрешены нужные API). Потом, даже с рабочим в браузере ключом, серверный вызов из lead.php падал - потому что «Application restrictions = HTTP referrers»: серверный запрос идёт БЕЗ referrer -> блок.
- **Как применять:** включить нужные API (Directions, Maps Static, Street View) И в ключе: «API restrictions» -> добавить эти API; «Application restrictions» -> **None или IP сервера**, НЕ «HTTP referrers». Проверка: открыть Directions-URL с ключом, ждать `status: OK`.

### 7.5 Встроенные картинки vs кнопки: кнопки надёжнее.
- **Почему:** встроенная карта/Street View - серверный вызов на каждой заявке (латентность + может молча падать: ключ / исходящий HTTP / хостинг). Кнопки «Открыть в Google Maps / Waze» - просто ссылки, всегда работают, открывают ЖИВУЮ карту (Street View внутри).
- **Как применять:** кнопки - базовый надёжный слой (делать всегда). Встроенные картинки - опциональный «вау», только если ключ и хостинг позволяют; при сомнениях хватает кнопок.

### 7.6 Waze не умеет маршрут «откуда->куда» через URL.
- **Почему:** Waze всегда ведёт от текущего GPS к точке; один URL не задаёт origin+destination.
- **Как применять:** две отдельные кнопки Waze (к «Откуда» и к «Куда»); полный маршрут - через кнопку Google Maps (`maps/dir/?api=1&origin=..&destination=..`).

## 8. Производительность (PageSpeed)

### 8.1 Не ставить лишние preload. Preload элементов ниже экрана крадёт канал у LCP.
- **Почему:** в `<head>` было 20 preload, из них 13 - логотипы партнёров (бегущая строка ниже экрана). На медленном 4G они отъедали канал у hero-картинки (это LCP) -> LCP 6.5s, Performance 72.
- **Как применять:** preload только для критичного над сгибом (hero + основные шрифты). Ниже-экранные картинки - лениво, без preload.

### 8.2 Кэш для SVG в .htaccess.
- **Почему:** `ExpiresByType` не включал `image/svg+xml` -> логотипы-SVG без кэша (варнинг PSI).
- **Как применять:** добавить `ExpiresByType image/svg+xml "access plus 1 year"`.

### 8.3 Лаб-троттлинг != реальные юзеры.
- **Почему:** мобильный Performance 72 - это лаб-тест под Slow 4G + слабый телефон, field-данных ещё нет. Десктоп и реальные CWV обычно лучше.
- **Как применять:** чинить очевидное (preload, размеры картинок, кэш), но помнить: для SEO важнее field-CWV живых юзеров, а A11y/BestPractices/SEO уже 97-100.

## Мобильная плавающая панель (mob-bar) — ТОЛЬКО кнопка «Позвонить»

**Правило (сверено с живым moving24.ee):** внизу на мобилке плавает **одна** кнопка — звонок (`btn-accent`, снизу-справа, с иконкой телефона), а НЕ две (звонок + заявка).

Почему:
- Две крупные кнопки (Позвонить + Заявка) перекрывают контент при скролле (налезают на заголовки/текст).
- moving24 держит одну кнопку звонка: `<a class="btn btn-accent mob-call" href="tel:…"><svg phone/>Helista</a>`.
- CSS (mobile): `.mob-bar{display:flex;justify-content:flex-end;pointer-events:none}` + `.mob-bar a{pointer-events:auto}` — кнопка прижата вправо и НЕ блокирует скролл сквозь панель.
- Обязателен JS на `visualViewport`: панель **прячется, когда пользователь печатает** в форме (focusin на input/textarea/select) и **поднимается над клавиатурой**. Без этого кнопка перекрывает поля формы на телефоне.

Заявку на мобилке даёт форма в hero + кнопки в секциях, отдельная плавающая кнопка «Заявка» не нужна.

**Общий принцип:** когда делаешь любой элемент — сверяйся с живым `moving24.ee` (это эталон), а не додумывай. Если поведение отличается от эталона — приведи к эталону и зафиксируй правило здесь.
