# Плейбук запуска под нового клиента

Цель: превратить «новый клиент в нише X» (например, компания по заборам) в готовый сайт, **клонируя Moving24**, с минимумом ручного проектирования. Меняется контент и структура под нишу - каркас, форма, трекинг остаются.

Главная идея: **сначала анализ, потом сборка**. Не все ниши ложатся на шаблон одинаково. Этот плейбук даёт процесс и оценку «насколько ниша отклоняется от стандарта Moving24», чтобы заранее знать, где клон почти 1-в-1, а где нужен кастом.

Инструмент данных: **Ahrefs MCP** (site-explorer, keywords-explorer, serp-overview и т.д.). Ниже в шагах указано, какой инструмент для чего.

---

## Шаг 0. Бриф бизнеса (10 минут с клиентом)

Собери минимум:
- **Услуги** (список того, что продают) и что из этого - «якорь» (главный спрос).
- **Гео**: один город / регион / вся страна / много городов. Это ключевой фактор структуры (см. city-pages).
- **Языки** (Moving24 = ET/RU/EN/FI; у заборов в Эстонии, вероятно, ET+RU, иногда EN).
- **Тип лида**: заявка формой, звонок, запрос замера/расчёта. Влияет на форму и CTA.
- **Средний чек / цикл сделки**: дорогая услуга с замером (заборы, ремонт) - CTA «бесплатный замер»; дешёвая/срочная (вывоз мусора) - «звоните сейчас».
- **Реальные данные**: телефон, email, адрес, рег/VAT, настоящие отзывы (Google/соцсети). Без них сайт не запускать (правило: не выдумывать - см. `LESSONS-FROM-BUILD.md`).

---

## Шаг 1. Найти конкурентов

Цель - 3-7 прямых конкурентов, которые уже ранжируются в этой нише и гео.

1. **Руками**: вбей в Google 3-5 главных запросов ниши по гео (напр. «aiad tallinn», «aia paigaldus», «piirdeaiad»). Выпиши, кто в топе органики (не агрегаторы, а такие же локальные бизнесы).
2. **Ahrefs - органические конкуренты**: по домену одного найденного конкурента запроси `site-explorer-organic-competitors` - Ahrefs покажет, кто пересекается по ключам. Добавь новых.
3. **Ahrefs - SERP**: `serp-overview` по главному ключу - увидишь топ выдачи + метрики страниц (что реально ранжируется).

Отбирай: локальные бизнесы того же типа (не маркетплейсы, не Википедию). 3-5 сильных - достаточно.

---

## Шаг 2. Снять данные по конкурентам (Ahrefs)

По каждому конкуренту-домену:

| Что нужно | Инструмент Ahrefs MCP | Зачем |
|---|---|---|
| **Топ-страницы по трафику** | `site-explorer-top-pages` | Понять, какие страницы приносят трафик - это готовая карта «какие страницы делать» |
| **Органические ключи** | `site-explorer-organic-keywords` | По каким запросам ранжируются - основа keyword-карты и текстов |
| **Объём/сложность ключей** | `keywords-explorer-overview`, `keywords-explorer-volume-by-country` | Приоритизировать: что имеет спрос в нужной стране |
| **Смежные/подсказки** | `keywords-explorer-related-terms`, `keywords-explorer-search-suggestions` | Найти под-темы, длинный хвост, темы для FAQ |
| **Объём метрик домена** | `site-explorer-metrics`, `site-explorer-domain-rating` | Оценить силу конкурента (сколько трафика, DR) |

Практика:
- Выгрузи топ-20 страниц каждого конкурента - сгруппируй по смыслу (услуги / гео / инфо / блог).
- Выгрузи топ-органик-ключи - сгруппируй в «кластеры» (одна страница = один кластер интента).
- **Тексты**: посмотри реально ранжирующиеся страницы конкурентов, забери СМЫСЛ и структуру (заголовки, блоки, FAQ), НЕ копипастом. Свой текст пишем по языковым плейбукам (`playbooks/`), нативно.

Итог шага 2: список «страниц, которые приносят трафик в нише» + keyword-карта (кластер -> запросы -> объём).

## Шаг 3. Предложить структуру сайта

Собери структуру из кластеров:
- **Услуги** (по одной странице на ядровую услугу) - как `teenused/*` у Moving24.
- **FAQ** (из подсказок и «people also ask») - как `kkk`.
- **О нас / доверие** - как `meist`.
- **Гео-страницы (city-pages)** - если ниша локальная и спрос гео-гранулярный (см. Шаг 4).
- **Легал** (privacy/terms) - как есть в шаблоне.

Каждой странице сразу назначь: целевой кластер, главный ключ, title/H1-набросок, язык(и).

---

## Шаг 4. City-pages / программные страницы - да или нет?

Это главная развилка отклонения от стандарта. Moving24 = **1 гео** («Таллин + вся Эстония»), без отдельных страниц по городам.

**Делать city-pages, если:**
- спрос гео-гранулярный (люди ищут «aiad + [город]», «забор + [район]»);
- бизнес реально работает по многим городам;
- конкуренты ранжируются по гео-страницам (видно в их top-pages из Шага 2).

**Как делать правильно (кратко; детали - `GUIDE.md`, секция про города):**
- URL-паттерн: `/услуга/[город]/` или `/[город]/услуга/`.
- **Уникальный контент под каждый город** (районы, местные ориентиры, местные кейсы/цены). Тонкие клоны-дубли = вред, Google их топит.
- `areaServed` в JSON-LD под город; внутренняя перелинковка город<->услуга; обновить sitemap + hreflang.
- Если городов много и контент шаблонизируется - это **программные страницы**: нужен генератор (в Moving24-репо есть Python-генераторы страниц - переиспользуй подход).

**Не делать**, если бизнес в одном городе или спрос не гео-гранулярный - тогда хватает одной страницы услуги + `areaServed` списком (как у Moving24).

## Шаг 6. Оценка «отклонения от стандарта»

По результату Шага 5 поставь оценку - она определяет объём работы:

**LOW (клон почти как есть, меняем только контент):**
- услуг <=5, city-pages нет (или пара), языки <=4, конверсия = форма+звонок, спец-функций нет.
- Пример: локальная услуга в одном гео (клининг «под ключ» в Таллине).
- Работа: тексты по плейбукам + картинки + реквизиты + перекраска + свой GTM/config. Дни.

**MEDIUM (клон + несколько кастомов):**
- есть city-pages (нужен генератор/несколько страниц), или нестандартные поля формы (замер/адрес объекта), или доп. блок (портфолио/галерея работ).
- Пример: **заборы** - услуги (типы заборов) + возможно city-pages по районам + форма с «тип забора / длина / замер».
- Работа: LOW + программные гео-страницы + кастом формы/блоков. 1-2 недели.

**HIGH (каркас берём, но много новой структуры):**
- каталог товаров, онлайн-бронирование/оплата, калькулятор, десятки услуг, сложная гео-сетка.
- Пример: аренда техники с каталогом и ценами, или сервис с онлайн-записью.
- Работа: шаблон = основа под лендинги/услуги, но ядро функционала - отдельная разработка. Оценивать индивидуально.

**Выведи в бриф:** оценка (LOW/MEDIUM/HIGH) + список того, что кастомим сверх стандарта.

## Шаг 7. Финальный «бриф ниши» перед сборкой

Один документ на проект (сохраняй рядом с сайтом клиента):
1. Бизнес и гео, языки, тип лида.
2. Конкуренты (3-5) + ссылки.
3. Keyword-карта (кластер -> ключи -> объём).
4. Предложенная структура (список страниц + главный ключ + title/H1).
5. City-pages: да/нет + паттерн.
6. Оценка отклонения LOW/MEDIUM/HIGH + список кастомов.
7. Чек реальных данных (телефон, email, адрес, рег/VAT, отзывы) - собраны ли.

Есть бриф -> запускаешь клон по `GUIDE.md`, тексты по `playbooks/`, не забываешь правила из `LESSONS-FROM-BUILD.md`.
